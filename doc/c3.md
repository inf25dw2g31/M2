# C3 : Product

## 3.1 Development

### Visão geral

O FestivAll é uma Single Page Application (SPA) construída em **ReactJS** que consome uma API REST própria com autenticação JWT e a API pública da Ticketmaster. Toda a aplicação está containerizada com Docker.

A app foi pensada como um conjunto de pequenos componentes React reutilizáveis, com gestão de estado em múltiplos níveis e proteção de rotas por autenticação.

---

### A escolha do React

O React foi a base de todo o frontend. Foram seguidos os princípios fundamentais da biblioteca:

**Componentes funcionais com Hooks**
Toda a aplicação usa exclusivamente componentes funcionais. Não existe uma única classe — todas as páginas, todos os componentes reutilizáveis e todos os providers são funções JavaScript que retornam JSX.

**JSX em vez de templates**
A interface é construída em JSX — uma extensão do JavaScript que parece HTML mas é processada como código. Isto permite misturar lógica condicional, loops e expressões diretamente na "vista" do componente, sem ter de saltar entre ficheiros HTML e JavaScript.

**Composição de componentes**
A app não é uma página gigante. É uma árvore de componentes pequenos que se compõem. O `Dashboard` contém um `Hero` + vários `EventCard` + um `Footer`. O `EventCard` contém uma `CardMedia` + várias `Typography` + um `Button`. Cada peça encaixa noutra.

**Hooks essenciais usados**

| Hook | Onde é usado |
|------|--------------|
| `useState` | Em quase todos os componentes — campos de formulário, listas, estados de loading |
| `useEffect` | Carregar dados da API quando uma página abre, sincronizar com localStorage |
| `useContext` | Aceder ao AuthContext e FavoritesContext em qualquer parte da app |
| `useMemo` | Memoizar valores derivados nos contextos para evitar re-renders desnecessários |
| `useNavigate` | Navegação programática após login/logout |
| `useLocation` | Detetar a rota atual para destacar o link ativo na NavBar |

---

### Estrutura de componentes criados

Foram desenvolvidos **15+ componentes próprios**, organizados por responsabilidade:

**Páginas (10)**
Login, Dashboard, Events, EventDetail, Artists, Favorites, History, Lists, Ratings, Profile.

Cada página é um componente funcional independente que se preocupa apenas com a sua própria lógica. Por exemplo, a `Events.jsx` gere o estado dos filtros de pesquisa, faz a chamada à API quando se submete o formulário, e renderiza os resultados num grid de `EventCard`.

**Componentes reutilizáveis (4)**
NavBar, Footer, EventCard, SearchBar.

O `EventCard` é o melhor exemplo do princípio de reutilização — o mesmo componente é usado em três páginas diferentes (Dashboard, Events e Favorites) recebendo o evento como prop. Foi escrito uma vez e é reutilizado em todos os locais.

**Providers de Context API (2)**
AuthProvider e FavoritesProvider.

São componentes especiais que envolvem toda a app no `main.jsx` e disponibilizam estado partilhado a qualquer descendente. Em vez de passar dados manualmente de componente em componente (prop drilling), os providers oferecem um "canal" acessível por hooks customizados (`useAuth`, `useFavorites`).

**Componentes estruturais (3)**
App, AppShell, RequireAuth.

O `App.jsx` define todas as rotas com `React Router`. O `AppShell` envolve as páginas com a NavBar comum e o Footer. O `RequireAuth` é um Higher-Order Component que protege rotas privadas — antes de renderizar a página pedida, verifica se o utilizador está autenticado, e em caso contrário redireciona para `/login`.

---

### React Router e navegação SPA

A navegação entre páginas usa o **React Router v6**. Em vez de fazer refresh ao mudar de página, o router intercepta os cliques e troca apenas o componente que está a ser renderizado, mantendo o resto da app (como a NavBar) imutável.

Cada rota está envolvida no `RequireAuth` excepto a `/login`. Isto garante que qualquer tentativa de acesso direto a uma URL protegida sem estar autenticado é automaticamente redirecionada para o login.

A NavBar usa `NavLink` em vez de `Link` para destacar visualmente qual é a página ativa — quando uma rota está ativa, o link recebe a classe `active` que aplica uma cor diferente.

### Material UI integrado

A interface foi construída combinando CSS próprio com a biblioteca **Material UI (MUI)**, integrada profundamente nos quatro componentes mais visíveis da aplicação.

**Porquê MUI?**
O MUI fornece componentes de interface prontos e profissionais, com acessibilidade incluída, animações de foco e estados, e total controlo de personalização através da prop `sx`. Em vez de escrever do zero um campo de input ou um botão com todos os estados (hover, focus, disabled, error), usa-se o componente do MUI e personaliza-se apenas o necessário.

**Componentes usados no FestivAll:**

No **Login** foram usados Paper, TextField, Button, Typography, Box e Alert. O `Paper` cria o painel branco com sombra, o `TextField` substitui os inputs HTML clássicos com label flutuante integrada, e o `Alert` mostra mensagens de erro de forma consistente.

Na **NavBar** foram usados AppBar, Toolbar, Button e Box. O `AppBar` é a barra superior fixa, o `Toolbar` alinha o conteúdo horizontalmente, e cada link é um `Button` com `component={NavLink}` — uma técnica que diz ao MUI para usar o NavLink do React Router por baixo, combinando o melhor dos dois mundos.

No **EventCard** foram usados Card, CardMedia, CardContent, Typography, Button, IconButton, Box e Chip. O `Card` substitui a `<article>` HTML com sombra e bordas arredondadas. O `CardMedia` gere a imagem com proporções corretas. O `Chip` exibe o status do evento como uma etiqueta. O `IconButton` é usado para o botão de favorito (estrela).

Na **SearchBar** foram usados TextField, Select, MenuItem, FormControl, InputLabel, Button e Box. O `Select` combinado com `FormControl` e `InputLabel` cria o dropdown de categoria com o mesmo estilo dos outros campos.

**Total de componentes MUI integrados: 14 diferentes**

**Personalização com `sx`**
Em vez de criar ficheiros CSS separados para o MUI, foi usada a prop `sx` para personalizar cada componente diretamente no JSX. Esta abordagem permite ter os estilos colocados junto do componente que estiliza, ler propriedades do tema, e fazer estilos condicionais baseados em props ou estado.

**Tema customizado**
No `theme.js` foi definido um tema MUI personalizado com:
- Paleta de cores azuis (primary, secondary)
- Tipografia (Inter como fonte principal)
- Border radius global (cantos arredondados consistentes)
- Estilos por defeito de componentes específicos

O tema é aplicado a toda a app através do `ThemeProvider` no `main.jsx`, garantindo coerência visual sem ter de repetir as cores em cada componente.

---

### Animações com Framer Motion

Para complementar a interface foram adicionadas animações declarativas com **Framer Motion**, uma biblioteca específica para React que torna animações complexas em algumas linhas de código.

**Três animações implementadas:**

1. **Pop no Login** — Quando a página de Login aparece, o painel inteiro faz um efeito de pop com transição tipo mola (spring). É implementado combinando o `Paper` do MUI com `motion.div` do Framer Motion através da prop `component`, conseguindo ter as features do MUI e as animações do Framer no mesmo elemento.

2. **Scale nos EventCard** — Quando o utilizador passa o rato sobre um card, ele cresce subtilmente 4% com transição spring. Usa a prop `whileHover` do Framer Motion, que automaticamente trata da animação de entrada e saída.

3. **Sublinhado animado na NavBar** — Quando o utilizador passa o rato sobre um link, aparece um sublinhado azul que expande do centro para os lados. Esta animação foi feita com CSS puro através de pseudo-elementos `::after` na prop `sx` do MUI, demonstrando que nem todas as animações precisam de bibliotecas externas.

### Gestão de estado em três níveis

A app gere estado em três níveis distintos, cada um adequado a um tipo de dados.

**Nível 1 — useState (estado local)**
Cada componente usa `useState` para os seus próprios dados temporários. A `Events.jsx` tem o estado dos filtros de pesquisa, a `Login.jsx` tem os campos do formulário, o `Dashboard.jsx` tem o estado de loading. Estes dados não precisam de ser partilhados com outros componentes.

**Nível 2 — Context API (estado partilhado)**
Para dados que múltiplas páginas precisam (autenticação e favoritos), foram criados dois contextos. O `AuthContext` guarda o token JWT, o utilizador atual e expõe funções `login`, `register`, `logout` e `updateApiKey`. O `FavoritesContext` mantém uma cópia local da lista de favoritos do utilizador, sincronizada com o backend, e oferece funções `toggleFavorite` e `isFavorite`.

Usar Context API evita o prop drilling — passar a mesma informação por vários níveis de componentes. Qualquer componente pode aceder ao token JWT chamando simplesmente `useAuth()`, sem importar quão profundo está na árvore.

**Nível 3 — Persistência (BD via API)**
Os dados que devem sobreviver a fechos do browser e logouts (favoritos, histórico, listas, avaliações) estão na base de dados MySQL. O frontend nunca acede diretamente à BD — todas as operações passam pela API REST. O token JWT também é guardado no `localStorage` para que o utilizador não tenha de fazer login a cada visita.

### Comunicação com APIs

Foi criado um módulo `src/api/backend.js` que centraliza todas as chamadas à API. Em vez de espalhar `fetch` por toda a app, este módulo expõe um objeto `api` com métodos semânticos como `api.login()`, `api.addFavorite()`, `api.getHistory()`, etc.

Cada método já trata automaticamente:
- A URL base correta (configurada via variável de ambiente)
- O header `Authorization: Bearer <token>` lido do localStorage
- O header `Content-Type: application/json`
- O parsing da resposta JSON
- A propagação de erros através de `throw new Error(...)`

Isto significa que as páginas usam a API com sintaxe muito limpa, por exemplo:

```js
const events = await api.getFavorites();
```

Existe também um módulo separado `src/api/ticketmaster.js` que faz chamadas à API pública da Ticketmaster, enviando a Consumer Key como query parameter.

## 3.2 Instalation

### Pré-requisitos

- Docker e Docker Compose instalados
- Conta no portal de developers da Ticketmaster ([developer.ticketmaster.com](https://developer.ticketmaster.com)) para obter a Consumer Key gratuita

### Execução em produção (recomendado)

Na raiz do projeto:

```bash
docker compose -f docker-compose-prod.yml up
```

Este comando puxa as imagens já publicadas no DockerHub e arranca os três serviços com um único comando. Não é necessário código-fonte para correr — apenas o `docker-compose-prod.yml`.

Aceder à aplicação:

```
http://localhost:3000
```

### Execução em desenvolvimento

Para quem queira compilar localmente com o código atual:

```bash
docker compose up --build
```

### Script com menu (Windows)

Foi incluído um script `festivall.bat` que oferece um menu interativo com opções para arrancar dev, arrancar prod, parar containers, ver logs, etc.

## 3.3 Usage

### Primeira utilização

1. Aceder a `http://localhost:3000`
2. Clicar em "Registar"
3. Preencher username, password e Consumer Key da Ticketmaster
4. Submeter — a app cria a conta, encripta a Consumer Key e devolve um JWT
5. O utilizador é redirecionado para o Dashboard

### Funcionalidades disponíveis

Após autenticado, o utilizador tem acesso a:

- **Dashboard** com eventos em destaque
- **Pesquisa de eventos** com filtros por palavra-chave, cidade, data e categoria
- **Detalhe de cada evento** com link para a bilheteira
- **Pesquisa de artistas**
- **Sistema de favoritos** persistente na base de dados
- **Histórico** de eventos consultados
- **Listas personalizadas** estilo playlists
- **Avaliações** com estrelas e notas
- **Perfil** com gestão da Consumer Key

## 3.4 Implementation Details

### Objetivos mínimos cumpridos

**Aplicação cliente em ReactJS** — App SPA construída com React 18 + Vite, com 15+ componentes funcionais, exclusivamente com Hooks. Sem componentes de classe, sem código legacy.

**Consumo de API REST com autenticação** — Foi implementado um backend próprio em Node.js + Express com autenticação JWT real. A API valida o token em cada pedido protegido através de middleware. A app também consome a API pública da Ticketmaster usando a Consumer Key do utilizador.

**Containerização com Docker** — Três serviços (web, api, db) orquestrados com Docker Compose. Foi criado um docker-compose-prod.yml com todas as imagens publicadas no DockerHub, permitindo executar a app sem precisar de código local.

**Submissão via GitHub e DockerHub** — Repositório versionado no GitHub, três imagens publicadas no DockerHub.

### Elementos de valorização

**Material UI em profundidade** — 14 componentes MUI integrados nos quatro componentes mais visíveis da app, com personalização via prop `sx` e tema customizado global.

**Animações declarativas com Framer Motion** — Três animações distintas que complementam o MUI: pop no login, scale no hover dos cards, sublinhado expansivo na navbar.

**Routing protegido com HOC** — Componente `RequireAuth` que protege todas as rotas privadas e redireciona automaticamente para login quando o utilizador não está autenticado.

**Backend próprio com 18 endpoints REST** — API completa em Node.js + Express com autenticação JWT, hash de passwords com bcrypt, encriptação simétrica da Consumer Key, rate limiting e cascade delete.

**Base de dados relacional** — 7 tabelas MySQL com chaves estrangeiras, constraints de unicidade, campos JSON e timestamps automáticos.

**Three-tier architecture com Docker Compose** — Frontend, API e base de dados como containers independentes, com healthchecks que garantem a ordem correta de arranque.

**Configuração Nginx para SPA** — Configuração específica que redireciona todas as rotas para `index.html`, permitindo que o React Router gira a navegação client-side mesmo em produção.

**Imagens publicadas no DockerHub** — Três imagens (`-react`, `m2`, `-mysql`) publicadas e acessíveis publicamente, permitindo correr a app em qualquer máquina sem código.

**Script de menu interativo (.bat)** — Ferramenta auxiliar para Windows que facilita a interação com os containers durante demonstrações.

**Funcionalidades extra além dos requisitos** — Histórico de eventos, sistema de avaliações, listas personalizadas, pesquisas guardadas e gestão de perfil com atualização da Consumer Key.
