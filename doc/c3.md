# C3 : Product

## 3.1 Development

### Arquitetura geral

O FestivAll segue uma arquitetura **three-tier** (três camadas), com responsabilidades bem separadas e comunicação por HTTP/REST entre cada camada.

```
┌─────────────────────────────────────────────┐
│   Cliente (Browser)                          │
│   React 18 + Vite + Material UI              │
│   localhost:3000                             │
└───────────────┬─────────────────────────────┘
                │ HTTPS / fetch
                ▼
┌─────────────────────────────────────────────┐
│   API REST (Backend)                         │
│   Node.js + Express + JWT                    │
│   localhost:4000                             │
└───────────────┬─────────────────────────────┘
                │ mysql2 driver
                ▼
┌─────────────────────────────────────────────┐
│   Base de Dados                              │
│   MySQL 8.0 (volume Docker persistente)      │
│   localhost:3306                             │
└─────────────────────────────────────────────┘
```

A app cliente também comunica com a **Ticketmaster Discovery API** (externa) para obter os dados públicos dos eventos.

### Padrões e princípios aplicados

**Separation of concerns** — Cada camada tem uma responsabilidade única. O frontend trata da apresentação, o backend da lógica de negócio e autenticação, a base de dados da persistência.

**Component composition** — A app é construída combinando componentes funcionais pequenos e reutilizáveis. O `EventCard` é usado em três páginas diferentes (Dashboard, Events, Favorites) sem duplicação de código.

**Stateless API** — A API não mantém estado entre pedidos. Cada pedido inclui o seu token JWT, permitindo escalabilidade horizontal.

**Single source of truth** — O estado partilhado (autenticação, favoritos) está em context providers, não duplicado em vários sítios.

**Defense in depth** — Múltiplas camadas de proteção (JWT + middleware + hash + encriptação + rate limit) em vez de uma única barreira.

## 3.2 Installation

### Pré-requisitos

- **Docker** e **Docker Compose** instalados
- Conta no portal de developers da Ticketmaster ([developer.ticketmaster.com](https://developer.ticketmaster.com)) para obter Consumer Key gratuita

### Execução com Docker (todos os serviços)

Na raiz do projeto:

```bash
docker compose up --build
```

Este comando faz:
1. Constrói a imagem do frontend (Node 20 → Nginx)
2. Constrói a imagem da API (Node 20)
3. Arranca o MySQL 8 com schema inicial
4. Arranca a API na porta 4000
5. Arranca o frontend Nginx na porta 3000

Aceder à aplicação em:
```
http://localhost:3000
```

### Para parar todos os serviços

```bash
docker compose down
```

Para apagar também os dados persistentes:

```bash
docker compose down -v
```

### Configuração dos serviços

O `docker-compose.yml` define três serviços:

```yaml
services:
  db:        # MySQL 8.0 → porta 3306, volume persistente
  api:       # Node + Express → porta 4000, depende de db
  web:       # React build → Nginx → porta 3000, depende de api
```

A comunicação entre containers usa nomes de serviço como hostname (`db`, `api`, `web`). O frontend exposto em `localhost:3000` envia pedidos para a API exposta em `localhost:4000`.

### Variáveis de ambiente

| Serviço | Variável | Valor por defeito |
|---------|----------|-------------------|
| db | MYSQL_ROOT_PASSWORD | root1234 |
| db | MYSQL_DATABASE | festivall |
| db | MYSQL_USER | festivall_user |
| db | MYSQL_PASSWORD | festivall1234 |
| api | JWT_SECRET | festivall_jwt_secret_2024 |
| api | ENCRYPTION_SECRET | festivall_enc_secret_2024 |
| web | VITE_API_URL | http://localhost:4000 |

### Configuração Nginx (SPA routing)

O Nginx usa a diretiva `try_files` para redirecionar todas as rotas para `index.html`, permitindo que o React Router gira a navegação no cliente:

```nginx
location / {
    try_files $uri $uri/ /index.html;
}
```

Sem isto, ao aceder diretamente a `localhost:3000/events`, o servidor procuraria um ficheiro `events` no disco e devolveria 404.

## 3.3 Usage

### Primeira utilização (registo)

1. Aceder a `http://localhost:3000`
2. Clicar em "Ainda não tens conta? Registar"
3. Preencher:
   - **Username** (à escolha)
   - **Password** (à escolha)
   - **Consumer Key da Ticketmaster** (obtida no portal de developers)
4. Submeter — a app guarda a Consumer Key encriptada e devolve um JWT
5. Utilizador é redirecionado para o Dashboard

### Utilizações seguintes (login)

1. Preencher username + password
2. App devolve o JWT e a Consumer Key desencriptada
3. Sessão fica válida 7 dias

### Funcionalidades disponíveis após login

- **Dashboard** — eventos em destaque carregados automaticamente
- **Eventos** — pesquisa avançada com filtros
- **Artistas** — pesquisa de atrações
- **Favoritos** — eventos guardados, persistentes na BD
- **Histórico** — eventos consultados recentemente
- **Listas** — criação de listas personalizadas
- **Avaliações** — atribuição de estrelas e notas a eventos
- **Perfil** — atualização da Consumer Key
- **Sair** — invalida a sessão local (token removido)

### Limitações

- Consumer Key tem limite de 5000 chamadas/dia (plano gratuito Ticketmaster)
- A API depende da disponibilidade externa da Ticketmaster
- Sem recuperação de password (não implementada nesta versão)
- Sem upload de avatar no perfil

## 3.4 Implementation Details

### Objetivos mínimos cumpridos

**Aplicação cliente em ReactJS**
A aplicação é uma Single Page Application em React 18, com componentes funcionais exclusivamente e React Hooks (useState, useEffect, useContext, useMemo). Inclui 15+ componentes próprios distribuídos por páginas, componentes reutilizáveis, providers de contexto e componentes estruturais (App, AppShell, RequireAuth).

**Consumo de API REST com autenticação e autorização**
A aplicação consome duas APIs REST. A API FestivAll (própria) implementa autenticação JWT com bcrypt para passwords e encriptação para a Consumer Key. A API Ticketmaster (externa) é acedida através da Consumer Key gerida pelo utilizador autenticado.

**Containerização com Docker**
Três serviços orquestrados com Docker Compose: frontend (build multistage Node→Nginx), API (Node 20 Alpine) e base de dados (MySQL 8 com volume persistente). Todos os serviços comunicam pela rede interna do Docker, com portas expostas no host apenas onde necessário.

**Submissão via GitHub e DockerHub**
Código-fonte versionado no GitHub. Imagens Docker publicadas no DockerHub.

### Elementos de valorização

**Backend próprio com Node.js + Express**
Não exigido pelo enunciado, mas implementado para garantir uma camada de autenticação/autorização real. Expõe 18 endpoints REST distribuídos por 6 routers.

**Autenticação JWT real**
Tokens emitidos no login, válidos 7 dias, transmitidos no header `Authorization: Bearer` em cada pedido protegido. Middleware `requireAuth` valida e injeta `req.userId` automaticamente.

**Encriptação simétrica da Consumer Key**
A Consumer Key da Ticketmaster nunca é guardada em texto puro na base de dados. É encriptada com uma chave secreta antes de persistir e desencriptada apenas quando devolvida ao cliente que se autenticou.

**Hash de passwords com bcrypt**
10 rounds de salt, prática padrão da indústria. Impossível recuperar a password original mesmo com acesso à base de dados.

**Rate limiting**
Endpoint `/auth` limitado a 20 pedidos por 15 minutos por IP (proteção contra força bruta). Restantes endpoints limitados a 100 pedidos por minuto.

**Base de dados relacional com 7 tabelas**
Schema bem normalizado, chaves estrangeiras, restrições de unicidade, cascade delete e armazenamento JSON para dados estruturados (event_data, filters).

**Material UI integrado em profundidade**
14 componentes MUI diferentes integrados nos quatro componentes-chave: Login (Paper, TextField, Button, Typography, Box, Alert), NavBar (AppBar, Toolbar, Button, Box), EventCard (Card, CardMedia, CardContent, Typography, Button, IconButton, Box, Chip) e SearchBar (TextField, Select, MenuItem, FormControl, InputLabel, Button, Box). Cada componente customizado via prop `sx` mantendo coerência visual com o tema azul-escuro.

**Animações com Framer Motion**
Três animações distintas: pop com transição spring no painel de login (mounting animation), scale 1.04 no hover dos cards de eventos (whileHover), e sublinhado expansivo nos links da navbar (CSS puro com pseudo-elemento ::after).

**Routing protegido**
Componente `RequireAuth` como Higher-Order Component. Verifica o estado de autenticação no AuthContext e redireciona para `/login` em caso de acesso não autorizado, antes mesmo de tentar renderizar a página protegida.

**Three-tier architecture com Docker Compose**
Frontend, API e base de dados completamente independentes, lançados com um único comando. Os serviços podem ser escalados ou substituídos individualmente sem afetar os restantes.

**Configuração Nginx para SPA**
Configuração específica para suportar client-side routing do React Router, com fallback para `index.html` em todas as rotas e cache otimizado para assets estáticos.

**Functionalidades avançadas implementadas**
Para além dos requisitos básicos: histórico de eventos consultados, listas personalizadas (estilo playlists), sistema de avaliações com estrelas e notas, e pesquisas guardadas — todas com persistência por utilizador na base de dados.
