# C2 : Resources

A aplicação consome **duas APIs REST** distintas que trabalham em conjunto: a Ticketmaster Discovery API (externa) para obter dados de eventos, e a API REST FestivAll (interna) que protege os dados pessoais do utilizador através de autenticação e autorização.

---

## 1. API FestivAll (interna)

API REST desenvolvida em Node.js + Express, com base de dados MySQL e autenticação JWT.

### Endpoint base
```
http://localhost:4000
```

### Autenticação

Implementada com **JSON Web Tokens (JWT)**. O fluxo é o seguinte:

1. Utilizador regista-se com `username`, `password` e Consumer Key da Ticketmaster
2. Backend hash da password com **bcrypt** e encripta a Consumer Key
3. Backend emite um **token JWT** (válido 7 dias) e devolve-o ao cliente
4. Cliente envia o token em todos os pedidos protegidos no header:

```
Authorization: Bearer <token>
```

5. Middleware `requireAuth` valida o token e injeta `req.userId` antes de prosseguir

### Recursos disponíveis

| Método | Endpoint | Proteção | Descrição |
|--------|----------|----------|-----------|

### Camadas de segurança

- **Autenticação** — JWT com chave secreta e expiração de 7 dias
- **Autorização** — middleware `requireAuth` em todos os endpoints sensíveis
- **Hash de passwords** — bcrypt com 10 rounds de salt
- **Encriptação simétrica** — Consumer Key da Ticketmaster encriptada antes de persistir
- **Rate limiting** — 20 pedidos / 15 min no `/auth`, 100 pedidos / min nos restantes
- **CORS** — configurado para aceitar pedidos do frontend
- **Cascade delete** — eliminação de utilizador remove automaticamente todos os seus dados

### Esquema da base de dados

```sql
users          → id, username, password_hash, tm_api_key_enc, created_at
favorites      → id, user_id (FK), event_id, event_name, event_data (JSON), created_at
history        → id, user_id (FK), event_id, event_name, event_data (JSON), viewed_at
saved_searches → id, user_id (FK), label, filters (JSON), created_at
ratings        → id, user_id (FK), event_id, stars (1-5), note, rated_at
lists          → id, user_id (FK), name, created_at
list_events    → id, list_id (FK), event_id, event_name, event_data (JSON), added_at
```

## 2. Ticketmaster Discovery API (externa)

API pública mantida pela Ticketmaster que disponibiliza dados de eventos culturais e desportivos em todo o mundo.

### Endpoint base
```
https://app.ticketmaster.com/discovery/v2
```

### Autenticação

Consumer Key obrigatória em todos os pedidos como query parameter:

```
?apikey=CONSUMER_KEY
```

Sem chave válida, a API responde com `401 Unauthorized`.

### Recursos utilizados

| Método | Endpoint | Descrição |
|--------|----------|-----------|
| GET | `/events.json` | Pesquisa eventos com filtros |
| GET | `/events/{id}.json` | Detalhe de evento específico |
| GET | `/attractions.json` | Pesquisa de artistas/atrações |

### Limitações

- 5000 chamadas/dia no plano gratuito
- Apenas leitura (GET)
- Sem identidade de utilizador final

---

## Como as duas APIs trabalham juntas

```
[Browser/Cliente React]
        │
        ├──→ POST /auth/login          → [API FestivAll] → [MySQL]
        │    ← JWT token
        │
        ├──→ GET /events.json?apikey=  → [Ticketmaster] → JSON eventos
        │    com a Consumer Key
        │
        └──→ POST /favorites           → [API FestivAll] → [MySQL]
             com Authorization: Bearer
```

A Ticketmaster fornece os dados públicos dos eventos. A API FestivAll gere os dados pessoais do utilizador (favoritos, histórico, listas, avaliações) com autenticação JWT.
